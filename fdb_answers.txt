Q1: FoundationDB separates the system into a Transaction Management System (TMS) and a Storage System (SS) to improve scalability and modularity. The TMS handles transaction coordination, version assignment, and conflict detection, while the SS is responsible for storing and serving data. This separation allows each layer to scale independently and reduces complexity by isolating responsibilities.
Q2: The Resolver is responsible for detecting read-write conflicts between concurrent transactions. This happens during the commit phase, after the transaction receives a commit version but before it is finalized. The Resolver checks whether any keys in the read set have been modified, ensuring serializability.
Q3: The read version defines a consistent snapshot of the database that a transaction reads from. All reads within the transaction use this version, so they see a stable view even if other transactions commit in the meantime. This mechanism is critical for providing strong consistency guarantees like strict serializability.
Q4: Snapshot reads do not participate in conflict detection, unlike normal reads. This means they will not cause conflicts or retries, but they may read slightly outdated data. They are useful for read-heavy or analytical workloads where performance is more important than strict consistency.
Q5: FoundationDB achieves cheap recovery because data is continuously replicated from LogServers to StorageServers during normal operation. Since StorageServers already maintain up-to-date data, recovery does not require replaying a redo log. Instead, the system can quickly recover by reassigning roles and continuing from the latest consistent state.
